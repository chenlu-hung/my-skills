# 自主派工政策 (auto-dispatch)

接到 coding 任務時，先評估是否該用 **dispatch skill** 委派給其他 model（codex / gemini / opencode），而不是自己做。這是要你主動判斷的，不需要我明示「派工」。

## 符合以下「全部」才派工
- 任務自足、機械性或可平行：大量 docstring、跨檔重構、批次測試修復、boilerplate、格式化、樣板生成。
- 能寫成一份**自足 brief**，不需要本對話的上下文就能完成。
- 且滿足其一：
  - 任務大、會吃掉大量 context；或
  - 有 2 個以上彼此獨立、可平行的子任務。

## 以下「不派工」，自己做
- 需要密集來回討論、需要本對話深層上下文、需要設計判斷。
- 觸及安全敏感程式碼，或無法寫成自足 brief。
- 任務小到自己幾分鐘內就能做完（派工的 overhead 不划算）。

## 選 worker
- **codex** — 高難度程式改動（最強、sandbox、無網路）。
- **gemini** — 大上下文閱讀 / 大型重構。
- **opencode** — 廉價機械雜務（免費）。

## 執行方式（自主程度：宣告後直接發）
1. 判斷該派工後，用**一行**宣告：派給哪個 worker、為什麼、預期跑幾分鐘。
2. 直接發工，**除非我當下喊停**。不必每次等我點頭。
3. 依 dispatch skill 流程：自足 brief → 必要時 worktree 隔離 → 跑 → **驗證 diff／測試（絕不盲目 merge）** → merge → 清理 → 回報。
4. worker 失敗（`ok: false`）就自己接手或改派，據實回報，不要隱藏。
